# Implementation Plan

- [ ] 1. 建立统一的签名模式编排骨架
- [ ] 1.1 定义标准化请求输入与兼容映射
  - 统一使用 `sign_mode`，并保留登录提供方等上下文参数。
  - 让入口层只处理输入归一化与基础参数校验，不再承担模式分支决策。
  - 为未知模式、缺失 URL、非法 method 等场景预留统一错误出口。
  - _Requirements: 1.1, 1.3, 1.4, 1.5, 3.5_
- [ ] 1.2 定义签名模式注册、可用性与错误模型
  - 建立所有签名模式共享的能力契约，覆盖模式标识、优先级、可用性检查、signer 会话解析和错误分类。
  - 明确“已配置可用”“需要登录”“未配置”“初始化失败”几类状态的边界。
  - 让后续新增模式只需注册能力，不需要修改入口调度逻辑。
  - _Requirements: 2.1, 2.4, 2.5, 4.1, 4.2, 6.4, 6.5_

- [ ] 2. 实现模式注册中心与共享运行时能力
- [ ] 2.1 实现显式选模与自动选模规则
  - 支持显式指定模式时只尝试该模式，不对失败场景自动回退。
  - 支持未指定模式时按优先级从当前可用模式中自动选择。
  - 在没有任何可用模式时返回统一的配置提示，并附带各模式可操作说明。
  - _Requirements: 1.2, 2.2, 2.3, 3.1, 3.2_
- [ ] 2.2 实现共享缓存与并发初始化控制
  - 对同一模式复用已初始化的运行时上下文，避免重复构建签名能力。
  - 对并发进入的相同模式初始化做合并，保证同一时刻只有一次昂贵初始化或登录流程。
  - 失败初始化不得污染后续重试，成功结果可被同进程复用。
  - _Requirements: 6.1, 6.2, 6.5_
- [ ] 2.3 (P) 提取通用支付装配能力
  - 将 signer 到支付执行器的装配提取为共享能力，统一注册支持的网络与支付 scheme。
  - 确保所有模式复用同一条 `402 -> 解析 -> 签名 -> 重试` 执行管线。
  - 保持现有本地私钥与 quick wallet 的支付语义一致。
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_

- [ ] 3. 迁移并隔离现有签名模式实现
- [ ] 3.1 (P) 迁移本地私钥模式为独立领域模块
  - 将本地私钥模式拆出为独立实现，负责检查私钥是否存在并构建本地 signer。
  - 保持该模式无远程依赖、无登录副作用，并可被自动选模直接选中。
  - 让本地私钥模式的缺失提示和错误文案独立于其他模式。
  - _Requirements: 2.4, 3.2, 4.1, 4.3_
- [ ] 3.2 迁移 quick wallet 模式为独立领域模块
  - 将 token 检查、远程钱包连接、登录补全和远程 signer 构建收敛到单独模式实现中。
  - 区分“已登录可直接使用”“基础配置完整但需要登录”“配置不足无法启动”三种状态。
  - 保留缺少 token 时允许在请求链路内触发 device flow 登录的现有行为。
  - _Requirements: 2.1, 2.5, 3.3, 3.4, 4.1, 4.3_
- [ ] 3.3 (P) 预留插件钱包模式接入边界
  - 预留新的模式标识、注册入口和 signer 能力接入点。
  - 保证未来新增插件钱包时，不需要改动入口编排和共享支付管线。
  - 将该模式保持为未就绪但可识别的占位实现，避免与已上线模式混淆。
  - _Requirements: 4.1, 4.2, 4.5, 5.5_

- [ ] 4. 完成入口工具改造与兼容迁移
- [ ] 4.1 接入新的模式解析与请求编排流程
  - 将入口工具改为调用统一注册中心完成模式选择、会话准备与支付执行。
  - 保持现有请求方法、请求体校验和最终响应格式不变。
  - 移除入口层中与具体模式耦合的分支、缓存与登录控制逻辑。
  - _Requirements: 1.1, 1.2, 3.1, 4.4, 4.5, 5.4_
- [ ] 4.2 更新工具输入说明与兼容行为
  - 在工具 schema 和描述中主推 `sign_mode`，明确默认自动选择策略。
  - 清理旧字段残留，确保调用侧统一使用 `sign_mode`。
  - 对未知模式、模式未就绪和无可用模式输出统一且可操作的提示。
  - _Requirements: 1.3, 1.4, 1.5, 2.3, 3.2, 3.5, 6.5_

- [ ] 5. 验证重构结果并补齐回归保护
- [ ] 5.1 (P) 补齐模式选择与错误处理测试
  - 覆盖显式选模、自动选模、未知模式、无可用模式、模式未就绪等核心分支。
  - 验证错误输出中包含调用方定位问题所需的模式信息与缺失项提示。
  - 重点覆盖仅使用 `sign_mode` 时的模式选择与归一化行为。
  - _Requirements: 1.2, 1.3, 1.4, 2.2, 2.3, 3.1, 3.2, 3.5, 6.3, 6.5_
- [ ] 5.2 (P) 补齐本地私钥与 quick wallet 模式回归验证
  - 验证本地私钥模式仍可完成现有 x402 支付重试链路。
  - 验证 quick wallet 在已有 token 时直接签名，在缺少 token 时可触发登录后继续执行。
  - 验证两种模式复用共享支付管线后仍保持既有行为一致。
  - _Requirements: 2.4, 2.5, 3.3, 3.4, 5.1, 5.4, 6.3_
- [ ] 5.3 补齐缓存与并发安全验证
  - 验证同一模式重复调用时会复用已初始化结果。
  - 验证同一模式并发初始化只执行一次昂贵准备流程。
  - 验证初始化失败后可以重新进入成功路径，不残留脏状态。
  - _Requirements: 6.1, 6.2, 6.3, 6.5_
